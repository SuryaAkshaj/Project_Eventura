## PART 2 — DOCKER SETUP

Create `docker-compose.yml` at the monorepo root with these exact services:

### PostgreSQL 16
- Image: `postgres:16-alpine`
- Port: `5432:5432`
- Environment: `POSTGRES_DB=eventura`, `POSTGRES_USER=eventura_user`, `POSTGRES_PASSWORD=eventura_secret`
- Volume: `postgres_data:/var/lib/postgresql/data` for persistence
- Health check: `pg_isready -U eventura_user -d eventura`
- Networks: `eventura_network`

### PgBouncer
- Image: `bitnami/pgbouncer:latest`
- Port: `6432:6432` (app connects to PgBouncer on 6432, NOT directly to PostgreSQL on 5432)
- Environment:
  - `POSTGRESQL_HOST=postgres`
  - `POSTGRESQL_PORT=5432`
  - `POSTGRESQL_DATABASE=eventura`
  - `POSTGRESQL_USERNAME=eventura_user`
  - `POSTGRESQL_PASSWORD=eventura_secret`
  - `PGBOUNCER_POOL_MODE=transaction` ← transaction mode for Prisma compatibility
  - `PGBOUNCER_MAX_CLIENT_CONN=100`
  - `PGBOUNCER_DEFAULT_POOL_SIZE=20`
- Depends on: postgres (healthy)
- Networks: `eventura_network`

### Redis 7
- Image: `redis:7-alpine`
- Port: `6379:6379`
- Command: `redis-server --appendonly yes --requirepass eventura_redis_secret`
- Volume: `redis_data:/data` for persistence
- Health check: `redis-cli -a eventura_redis_secret ping`
- Networks: `eventura_network`

### Important notes:
- The `DATABASE_URL` in `.env` must point to PgBouncer (`localhost:6432`), NOT PostgreSQL directly (`localhost:5432`)
- The `DIRECT_URL` in `.env` must point to PostgreSQL directly (`localhost:5432`) — Prisma needs this for migrations since PgBouncer transaction mode does not support DDL statements
- Both URLs are required in `schema.prisma` for Prisma to work correctly with PgBouncer

## PART 4 — POSTGRESQL ROW LEVEL SECURITY

After running `prisma migrate dev`, run this raw SQL migration file to enforce RLS at the database level. Create `prisma/migrations/rls_policies.sql`:

```sql
-- Enable RLS on all tenant-scoped tables
ALTER TABLE "Club" ENABLE ROW LEVEL SECURITY;
ALTER TABLE "RoleAssignment" ENABLE ROW LEVEL SECURITY;
ALTER TABLE "Event" ENABLE ROW LEVEL SECURITY;
ALTER TABLE "EventSession" ENABLE ROW LEVEL SECURITY;
ALTER TABLE "Registration" ENABLE ROW LEVEL SECURITY;
ALTER TABLE "ScanLog" ENABLE ROW LEVEL SECURITY;
ALTER TABLE "Waitlist" ENABLE ROW LEVEL SECURITY;

-- RLS Policies — tenant isolation using app.current_college_id session variable
CREATE POLICY club_isolation ON "Club"
  USING ("collegeId"::text = current_setting('app.current_college_id', true));

CREATE POLICY role_assignment_isolation ON "RoleAssignment"
  USING ("collegeId"::text = current_setting('app.current_college_id', true));

CREATE POLICY event_isolation ON "Event"
  USING (
    "collegeId"::text = current_setting('app.current_college_id', true)
    OR "visibility" = 'PUBLIC'
    OR "visibility" = 'ALL_PLATFORM'
    OR EXISTS (
      SELECT 1 FROM "SharedEvent" se
      WHERE se."eventId" = "Event"."id"
      AND se."collegeId"::text = current_setting('app.current_college_id', true)
    )
  );

CREATE POLICY registration_isolation ON "Registration"
  USING (
    "userId"::text = current_setting('app.current_user_id', true)
    OR EXISTS (
      SELECT 1 FROM "Event" e
      WHERE e."id" = "Registration"."eventId"
      AND e."collegeId"::text = current_setting('app.current_college_id', true)
    )
  );

-- Super Admin bypass — set app.is_super_admin = 'true' to bypass all RLS
CREATE POLICY super_admin_bypass_clubs ON "Club"
  USING (current_setting('app.is_super_admin', true) = 'true');

CREATE POLICY super_admin_bypass_events ON "Event"
  USING (current_setting('app.is_super_admin', true) = 'true');
```

### `src/config/env.ts`
Use **Zod** to validate all environment variables at startup. If any required variable is missing the app must crash immediately with a clear error message listing what is missing. Required variables:
```

RESEND_API_KEY, CLIENT_URL, NODE_ENV, PORT
```

### `src/config/database.ts`
Implement the multi-tenant Prisma client using `AsyncLocalStorage` and `$extends` exactly as specified in the attached middleware specification document. The extended client must:
- Automatically inject `collegeId` into all `findMany`, `findFirst`, `update`, `delete` operations for tenant-scoped models
- Automatically inject `collegeId` into all `create` operations
- Skip tenant injection for: `College`, `Role`, `Permission`, `PlatformSettings`, `AuditLog` models
- Skip tenant injection entirely when `role === 'SUPER_ADMIN'`
- Export both `prisma` (extended, tenant-aware) and `prismaAdmin` (raw, Super Admin only)

### `src/config/redis.ts`
Create a Redis client singleton using `ioredis` with:
- Connection to `REDIS_URL` with password authentication
- Automatic reconnection with exponential backoff (max 10 retries)
- Connection event logging (connected, error, reconnecting)
- Export helper functions:
  - `setWithExpiry(key, value, ttlSeconds)` 
  - `getAndDelete(key)` — atomic get + delete for nonce validation
  - `setNX(key, value, ttlSeconds)` — atomic set-if-not-exists for double-scan prevention

### `src/shared/utils/apiResponse.ts`
Standardised response wrapper for ALL API responses:
```typescript
// Success
{ success: true, data: T, message?: string }

// Error  
{ success: false, error: { code: string, message: string, details?: any } }

// Paginated
{ success: true, data: T[], meta: { total, page, limit, totalPages } }
```

### `src/shared/utils/asyncHandler.ts`
Wrap all async route handlers to catch errors and pass to the global error handler. No try-catch blocks in controllers.

### `src/shared/constants/permissions.ts`
Define all permission strings as constants:

### `src/middleware/auth.middleware.ts`
JWT verification middleware that:
- Extracts Bearer token from `Authorization` header
- Verifies signature using `JWT_SECRET`
- Checks Redis blacklist — reject if token is blacklisted
- Checks `RoleAssignment.expiresAt` — reject if Event Manager role has expired
- Attaches `req.user` with full JWT payload including `activeContext`
- Sets `AsyncLocalStorage` tenant context for Prisma

### `src/middleware/rbac.middleware.ts`
Role-based access control middleware:
- `requireRole(...roles: RoleName[])` — checks if user has any of the required roles
- `requirePermission(...permissions: string[])` — checks granular permissions from JWT
- `requireEventAccess(role)` — validates Event Manager has access to the specific eventId in the route params

### `src/middleware/rateLimiter.middleware.ts`
Redis-backed rate limiter:
- General API: 100 requests per minute per IP
- Auth endpoints: 10 requests per minute per IP (stricter for brute force prevention)
- QR scan endpoint: 30 scans per minute per Event Manager (prevents scanner abuse)
- Returns `429 Too Many Requests` with `Retry-After` header

### `src/app.ts`
Express app with this exact middleware chain in order:
1. `helmet()` — security headers
2. `cors()` — configured with `CLIENT_URL` whitelist
3. `express.json({ limit: '10mb' })`
4. `requestLogger` — Morgan with unique request ID per request
5. `rateLimiter` — general rate limiting
6. Route modules (to be added in future missions)
7. `errorHandler` — global error handler (last middleware)
